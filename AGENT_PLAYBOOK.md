# AGENT_PLAYBOOK.md — 코딩 에이전트 작업 방법론

이 문서는 프로젝트 지식이 아니라 **작업하는 방식 그 자체**다. 어떤 모델(Opus, Codex, 기타)이든
이 문서의 규칙대로 움직이면 같은 품질의 결과가 나오도록 작성했다. 매 세션 시작 시 읽어라.
프로젝트 고유 지식은 `HANDOFF.md`와 `CLAUDE.md`에 있다.

---

## 원칙 1 — 증거 없이 고치지 마라

- 버그 보고를 받으면 **첫 행동은 수정이 아니라 관찰**이다: 로그를 읽고, 상태를 curl/스크린샷으로 확인하고, 재현한다.
- "아마 이것 때문일 것"이라는 가설은 반드시 확인 후에 수정으로 넘어간다. 가설이 두 개면 둘 다 확인하고 하나를 증거로 소거한다.
- 이 프로젝트의 관찰 도구: `~/.sketchupshow/debug.log`(플러그인), 브라우저 콘솔, `curl localhost:9876/api/ping`(브릿지 생사), dev 서버 실화면.
- 원인을 찾았으면 **사용자에게 원인을 한 문장으로 먼저 말하고** 고쳐라. "고쳤습니다"보다 "원인은 X였고, 이렇게 고쳤습니다"가 신뢰를 만든다.

## 원칙 2 — 증상이 아니라 근본 원인을 고쳐라

- 임시방편(workaround)·눈속임 금지. 예: "재질 텍스처가 안 나온다" → 화면 크롭으로 흉내내기(❌) vs 단일 재질 추출 API를 추가해 실제 텍스처를 가져오기(✅). 사용자는 눈속임을 반드시 알아챈다.
- 근본 수정이 크고 당장 사용자가 아프면: **즉시 완화 조치 + 근본 수정을 같이** 적용한다. (예: 무효 깊이맵 → ① 웹앱에서 자동 폐기[즉시 효과] ② 플러그인 캡처 버그 수정[근본]) — 그리고 두 가지를 모두 했다고 보고한다.
- 수정 후 "이 수정이 왜 문제를 없애는가"를 스스로 설명할 수 없으면 아직 근본 원인이 아니다.

## 원칙 3 — 검증 사다리를 끝까지 올라간 뒤에만 "됐다"고 말하라

순서대로, 하나라도 건너뛰지 마라:

1. **타입체크**: `npx tsc --noEmit` (CLI 결과만 신뢰. IDE 진단은 자주 낡았다)
2. **빌드**: `npm run build`
3. **실행 검증**: 브라우저에서 실제 클릭으로 기능을 끝까지 사용해본다. 상태 주입이 필요하면 DEV 전용 `window.__classicStore`/`__graphStore` 사용
4. **보고**: 무엇을 눈으로 확인했는지 명시 ("스크린샷으로 확인", "curl로 응답 확인")

- 검증 못 한 부분이 있으면 **명시적으로 남긴다**: "Blender 쪽은 실기 검증 전입니다."
- 실패·에러는 숨기지 말고 출력 그대로 보고한다. 테스트가 깨졌으면 깨졌다고 말한다.

## 원칙 4 — 전역 상태를 만지면 반드시 원상복구하라

- 렌더 옵션, 뷰포트 설정, 셰이딩 모드 등 **전역/공유 상태를 임시로 바꾸는 코드는 무조건 try/finally(ensure)로 전부 복원**한다. 저장→변경→작업→복원 패턴. 키별 실패는 개별 rescue로 감싸 전체 복원이 중단되지 않게 한다.
- 복원 누락은 "사용자 화면이 이상해졌다"는 별개 버그로 돌아온다.

## 원칙 5 — 외부에서 온 데이터는 검증하고 나서 써라

- API 응답, 캡처 이미지, 파일 등 외부 산출물은 **쓰기 전에 유효성 검사**를 넣는다. 예: 깊이맵의 명암 분포 검사 — 정보가 없는 입력을 AI에 넣으면 없느니만 못한 결과(환각)가 나온다.
- 검증 실패 시: 조용히 죽지 말고 **폴백 + 로그 + 사용자 안내**. 반복 실패는 카운터로 감지해 자동으로 그 경로를 끄고, 회복 조건도 함께 설계한다.

## 원칙 6 — 병렬 작업자(다른 에이전트/사람)를 전제로 움직여라

- **Edit 직전에 파일을 다시 읽어라.** 몇 분 전의 내용은 이미 낡았을 수 있다.
- **완성 즉시 커밋해라.** 커밋 안 된 작업물은 다른 에이전트의 저장 한 번에 증발한다.
- 남의 변경은 **절대 되돌리지 말고 흡수**해라. 빌드가 남의 미완성 코드로 깨지면 잠깐 기다렸다 재확인.
- 화면이 통째로 죽었는데 내 변경이 원인 같지 않으면: 콘솔에서 미정의 심볼 에러부터 확인 (동시 편집 중간 상태일 확률이 높다).

## 원칙 7 — 편집은 외과수술처럼

- 큰 파일 전체 재작성 금지. **앵커 문자열 기반의 정확한 부분 치환**으로 수정하고, 치환 전 앵커 존재를 assert로 확인해라 (없으면 파일이 바뀐 것 — 다시 읽어라).
- 주변 코드의 스타일·주석 밀도·네이밍을 따라간다. 주석은 코드가 말할 수 없는 제약("~하면 크래시", "~는 반대 방향")만 적는다.
- 새 기능은 기존 패턴을 먼저 찾아 재사용한다 (예: 새 캡처 기능 = 기존 capture_mask의 저장/복원 패턴 복제).

## 원칙 8 — 사용자 보고를 해석하는 법

- 사용자는 화면에서 본 것을 말한다. **사용자의 화면 기준으로 재현 경로를 상상**하고, 그 경로의 상태 흐름을 추적해라.
- "갑자기 느려졌다/이상해졌다" = **최근에 들어간 변경부터 의심**해라. 특히 자기 자신이 넣은 기능 (예: 깊이맵 캡처가 렌더마다 몇 초씩 낭비).
- 재현이 안 되면 상태를 주입해서라도 그 상황을 만들어본다.
- 모호한 요청은 가장 그럴듯한 해석으로 진행하되, **어떤 해석으로 했는지 결과 보고에 명시**한다. 사용자를 중간에 세워두는 질문은 최소화.

## 원칙 9 — 보고는 결론부터, 짧게, 정직하게

- 첫 문장 = 결과 ("완성했습니다", "원인은 X입니다"). 과정은 그 다음.
- 한 일 / 확인한 것 / 남은 것 세 가지를 구분해서 말한다.
- 사용자에게 수동 조치가 필요하면(재시작 등) **가장 쉬운 형태 하나로** 안내한다. 콘솔 명령 3줄보다 "재시작 한 번"이 낫다 — 복잡한 안내는 절반만 실행된다.

## 원칙 10 — 커밋 규율

- 파일 수정 후 즉시 커밋 (이 프로젝트는 사용자가 자동 커밋을 요구함). 커밋 전 검증 사다리 1·2단은 필수.
- 커밋 메시지 = 무엇을 + **왜** (원인/맥락 포함). "fix bug" 금지.
- `git checkout/reset/restore`는 커밋 해시 명시 + 사용자 허락 후에만.
- 배포 한도(Vercel 등)로 배포가 막혀도 커밋·푸시는 계속한다.

## 원칙 11 — AI 엔진을 다루는 법 (이 제품 특유)

- 생성 AI에게 "하지 마라"는 지시는 **참고사항일 뿐이다**. 결과를 신뢰하지 말고 **코드로 강제**해라 (마스크 영역 밖 = 원본 픽셀로 강제 합성).
- 참조 이미지를 여러 장 보낼 때는 배열 순서와 프롬프트의 "image N" 번호가 **기계적으로 일치**하도록 조립 코드를 한 곳에 모아라.
- API 기본값을 확인해라. 명시 안 하면 최저 품질이 기본인 경우가 있다 (xAI 영상 480p 사건).
- 프롬프트 지시는 [SECTION] 블록으로 구조화하고, 절대 조건은 CRITICAL/EXACTLY 같은 강한 어휘로.

## 원칙 12 — 막혔을 때

- 같은 시도를 3번 반복하지 마라. 접근을 바꾸거나, 더 낮은 레벨에서 관찰하거나(로그 추가, 최소 재현), 사용자에게 지금까지의 증거와 함께 상황을 보고해라.
- "모른다"는 보고도 유효한 결과다 — 단, 무엇을 확인했고 무엇이 배제됐는지 함께.
